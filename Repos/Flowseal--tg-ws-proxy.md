## 專案簡介

tg-ws-proxy 是一個本地的 MTProto 代理伺服器，旨在透過 WebSocket 連接來加速 Telegram Desktop 的使用體驗。它的工作流程是：Telegram Desktop 連接到本地代理（127.0.0.1:1443），代理再透過 WebSocket 連接到 Telegram 的數據中心（DC）。這種設計的好處在於，所有數據仍然保持加密狀態，並且不需要依賴外部伺服器。使用者可以透過簡單的 CLI 指令 `tg-ws-proxy` 啟動代理，並可選擇性地設定端口、主機和數據中心 IP。這個工具的核心技術使用了 Python，並且依賴於一些輕量級的庫，如 `pyperclip` 和 `psutil`，使得整體架構相對簡單且易於維護。

與其他代理工具相比，如 AlpinDale/parsync 和 BigBodyCobain/Shadowbroker，tg-ws-proxy 的優勢在於其專注於 Telegram 的 MTProto 協議，並且能夠自動切換到 TCP 連接，當 WebSocket 連接失敗時。實際使用中，這個工具能夠處理多個數據中心的連接，並且支持詳細的日誌記錄和配置選項，這對於需要監控流量的使用者來說非常有用。這個專案目前處於穩定階段，並且有活躍的社群支持，解決率達到 57%。對於需要在 Windows、macOS 或 Linux 上使用 Telegram 的用戶，tg-ws-proxy 提供了一個方便的解決方案。未來幾個月，隨著 Telegram 的使用需求增加，這個專案可能會進一步增強其功能和穩定性。

**技術棧**：`Python 3.8+` · `Dockerfile`

## 重點功能

- MTProto 代理 — 提供 Telegram 的本地代理服務，支持 WebSocket 連接。
- 自動切換 — 當 WebSocket 連接失敗時，能自動切換到 TCP 連接，確保連接穩定。
- 詳細日誌 — 支持詳細的日誌記錄，方便用戶監控和排查問題。
- 多平台支持 — 提供 Windows、macOS 和 Linux 的安裝包，方便不同用戶使用。
- CLI 支持 — 提供命令行介面，支持多種配置選項，如端口、主機和數據中心 IP。

## 架構分析

tg-ws-proxy 採用單一的 Python 應用架構，將 MTProto 代理與 WebSocket 連接整合在一起。這樣的設計使得應用能夠在本地快速啟動，並且能夠有效地處理 Telegram 的數據流。資料流經過 MTProto 代理後，會被轉換為 WebSocket 格式，這樣可以利用現有的網路基礎設施。這種設計的代價在於，使用者需要在本地運行代理，無法作為遠端服務使用。擴展性方面，這個架構能夠支持多個數據中心的連接，但在高流量情況下可能會遇到性能瓶頸。

> # TG WS Proxy
> 
> **Локальный MTProto-прокси** для Telegram Desktop, который **ускоряет работу Telegram**, перенаправляя трафик через WebSocket-соединения. Данные передаются в том же зашифрованном виде, а для работы не нужны сторонние сервера.
> 
> ## Как это работает
> 
> ```
> Telegram Desktop → MTProto Proxy (127.0.0.1:1443) → WebSocket → Telegram DC
> ```
> 
> 1. Приложение поднимает MTProto прокси на `127.0.0.1:1443`
> 2. Перехватывает подключения к IP-адресам Telegram
> 3. Извлекает DC ID из MTProto obfuscation init-пакета
> 4. Устанавливает WebSocket (TLS) соединение к соответствующему DC через домены Telegram
> 5. Если WS недоступен (302 redirect) — автоматически переключается на прямое TCP-соединение
>

> 
> | Аргумент | По умолчанию | Описание |
> |---|---|---|
> | `--port` | `1443` | Порт прокси |
> | `--host` | `127.0.0.1` | Хост прокси |
> | `--secret` | `random` | 32 hex chars secret для авторизации клиентов |
> | `--dc-ip` | `2:149.154.167.220`, `4:149.154.167.220` | Целевой IP для DC (можно указать несколько раз) |
> | `--buf-kb` | `256` | Размер буфера в КБ
> | `--pool-size` | `4` | Количество заготовленных соединений на каждый DC
> | `--log-file` | выкл. | Путь до файла, в который сохранять логи 
> | `--log-max-mb` | `5` | Максимальный размер файла логов в МБ (после идёт перезапись)
> | `--log-backups` | `0` | Количество сохранений логов после перезаписи
> | `-v`, `--verbose` | выкл. | Подробное логирование (DEBUG) |
> 
> **Примеры:**
> 
> ```bash
> # Стандартный запуск
> tg-ws-proxy
> 
> # Другой порт и дополнительные DC
> tg-ws-proxy --port 9050 --dc-ip 1:149.154.175.205 --dc-ip 2:149.154.167.220
> 
> # С подробным логированием
> tg-ws-proxy -v
> ```

> ## Конфигурация
> 
> Tray-приложение хранит данные в:
> 
> - **Windows:** `%APPDATA%/TgWsProxy`
> - **macOS:** `~/Library/Application Support/TgWsProxy`
> - **Linux:** `~/.config/TgWsProxy` (или `$XDG_CONFIG_HOME/TgWsProxy`)
> 
> ```json
> {
>   "host": "127.0.0.1",
>   "port": 1443,
>   "secret": "...",
>   "dc_ip": [
>     "2:149.154.167.220",
>     "4:149.154.167.220"
>   ],
>   "verbose": false,
>   "buf_kb": 256,
>   "pool_size": 4,
>   "log_max_mb": 5.0,
>   "check_updates": true
> }
> ```
> 
> Ключ **`check_updates`** — при `true` при запросе к GitHub сравнивается версия с последним релизом (только уведомление и ссылка на страницу загрузки). На Windows в конфиге может быть **`autostart`** (автозапуск при входе в систему).
