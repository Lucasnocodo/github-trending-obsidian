## 專案簡介

Mercury 是一個以「靈魂驅動」的 AI 代理，具備記憶、權限管理和多通道訪問功能。用戶可以透過 CLI 或 Telegram 與其互動，並使用 31 種內建工具。其核心機制是透過 SQLite 支持的第二大腦記憶系統，能夠記錄和回憶用戶的偏好和行為。安裝後，使用者只需執行 `mercury up` 命令即可啟動持久化的背景守護進程，並自動處理崩潰重啟。這個設計使得 Mercury 能夠在用戶不干預的情況下持續運行，並能夠在 Telegram 上進行即時交互。其「權限加固」的特性確保了安全性，防止未經授權的操作。

這個工具的賣點在於其能夠在多種 LLM 提供者之間自動切換，並且能夠根據用戶的需求進行自我學習和調整。相較於其他 AI 代理，Mercury 不僅能夠執行命令，還能在執行前詢問用戶，這樣的設計使得用戶對 AI 的行為有更高的控制權。與傳統的 AI 代理相比，Mercury 的記憶系統和靈魂定義使其更具個性化和靈活性。使用者可以透過簡單的 CLI 命令來管理其功能和設定，並且可以輕鬆地擴展其技能。這些特性使得 Mercury 成為一個適合需要持續運行和個性化交互的用戶的理想選擇。

**技術棧**：`TypeScript 5.7.0` · `Node.js 20.0.0` · `SQLite`

## 重點功能

- 權限加固 — 透過 Shell 阻擋清單和待批准流程，確保安全執行。
- 第二大腦記憶 — 使用 SQLite 支持的結構化記憶系統，能夠自動提取和記錄用戶信息。
- 靈魂驅動 — 透過 Markdown 文件定義個性，提供更人性化的交互體驗。
- 多通道支持 — 同時在 CLI 和 Telegram 上運行，並能夠進行即時交互。
- 自動重啟 — 在崩潰後自動重啟，並支持系統服務安裝，確保持續運行。

## 架構分析

Mercury 採用模組化架構，核心功能由 TypeScript 實現，並使用 SQLite 作為記憶存儲。這樣的設計使得記憶系統能夠快速查詢和更新，並且支持多種 LLM 提供者的自動切換。資料流從用戶輸入經過權限檢查後，執行相應的命令，並將結果返回給用戶。選擇 SQLite 作為記憶存儲的好處在於其輕量和高效，但在處理大量資料時可能會遇到性能瓶頸。擴展性方面，Mercury 支持安裝社群技能，這使得用戶可以根據需求擴展功能，但也可能增加維護的複雜度。

## 技術深入分析

Mercury 的核心技術機制包括使用 TypeScript 實現的模組化架構，並結合 SQLite 作為記憶存儲。這使得其能夠快速查詢和更新記憶資料，並支持多種 LLM 提供者的自動切換。效能方面，Mercury 能夠處理中小型資料集，但在面對大量資料時可能會出現性能瓶頸。選擇 SQLite 而非其他資料庫的原因在於其輕量和易於部署，但這也意味著在高並發的場景下可能會遇到挑戰。設計上，Mercury 的權限管理和靈魂驅動特性使其在安全性和個性化方面有明顯優勢。技術風險方面，隨著用戶數量的增加，記憶系統的性能可能會成為瓶頸，尤其是在高頻率的請求下。整合方面，Mercury 與主流的 CI/CD 工具相容性良好，並且能夠輕鬆集成到現有的開發流程中。

> [!info]- Release Notes
> ## Fix Ollama Cloud Provider
> 
> Ollama Cloud was completely broken — every request returned `404 Not Found`. Four independent bugs prevented `ollamaCloud` from functioning:
> 
> ### Bug 1: Wrong SDK
> `ollamaCloud` was routed through `OllamaProvider` which uses `ollama-ai-provider` — a package designed for **local** Ollama servers (`/api/chat`, `/api/tags`). Ollama Cloud exposes an **OpenAI-compatible** API at `/v1/chat/completions` and `/v1/models`.
> 
> **Fix**: `ollamaCloud` now routes through `OpenAICompatProvider` (using `createOpenAI()` from `@ai-sdk/openai`).
> 
> ### Bug 2: Wrong default base URL
> Default `OLLAMA_CLOUD_BASE_URL` was `https://ollama.com/api` (local server path). Correct URL is `https://ollama.com/v1`.
> 
> **Fix**: Updated default + added config migration that auto-upgrades existing configs from `/api` to `/v1`.
> 
> ### Bug 3: Responses API instead of Chat Completions
> After fixing Bug 1, `OpenAICompatProvider` defaulted to OpenAI's **Responses API** (`/responses`). Ollama Cloud only supports **Chat Completions** (`/chat/completions`).
> 
> **Fix**: Added `useChatApi` option to `OpenAICompatProvider`. `ollamaCloud` now uses `client.chat(model)`.
> 
> ### Bug 4: No baseUrl validation
> `isProviderConfigured()` only checked `apiKey` for `ollamaCloud` — missing `baseUrl` caused cryptic failures.
> 
> **Fix**: Added explicit validation for both `apiKey` and `baseUrl`.

> ## Why Mercury?
> 
> Every AI agent can read files, run commands, and fetch URLs. Most do it silently. **Mercury asks first — and remembers what matters.**
> 
> - **Permission-hardened** — Shell blocklist (`sudo`, `rm -rf /`, etc. never execute). Folder-level read/write scoping. Pending approval flow. Ask Me or Allow All per session. No surprises.
> - **Second Brain** — Persistent, structured memory with SQLite + FTS5 full-text search. 10 memory types, auto-extraction, conflict resolution, auto-consolidation. Mercury learns your preferences, goals, and habits without manual entry.
> - **Soul-driven** — Personality defined by markdown files you own (`soul.md`, `persona.md`, `taste.md`, `heartbeat.md`). No corporate wrapper.
> - **Token-aware** — Daily budget enforcement. Auto-concise when over 70%. `/budget` command to check, reset, or override.
> - **Live streaming** — Real-time token streaming on CLI with cursor-save/restore and markdown re-rendering. Telegram streaming with editable status messages.
> - **Always on** — Run as a background daemon on any OS. Auto-restarts on crash. Starts on boot. Cron scheduling, heartbeat monitoring, and proactive notifications.
> - **Extensible** — Install community skills with a single command. Schedule skills as recurring tasks. Based on the [Agent Skills](https://agentskills.io) specification.
> 
> 
> ## Daemon Mode
> 
> **One command to make Mercury persistent:**
> 
> ```bash
> mercury up
> ```
> 
> This installs the system service (if not installed), starts the background daemon, and ensures Mercury is running. Use this as your go-to command.
> 
> If Mercury is already running, `mercury up` just confirms it and shows the PID.
> 
> 
> ### Other daemon commands
> 
> ```bash
> mercury restart      # Restart the background process
> mercury stop         # Stop the background process
> mercury start -d     # Start in background (without service install)
> mercury logs         # View recent daemon logs
> mercury status       # Show if daemon is running
> ```
> 
> Daemon mode includes built-in crash recovery — if the process crashes, it restarts automatically with exponential backoff (up to 10 restarts per minute).
> 
> 
> ### System Service (auto-start on boot)
> 
> `mercury up` installs this automatically. You can also manage it directly:
> 
> ```bash
> mercury service install
> ```
> 
> | Platform | Method | Requires Admin |
> |----------|--------|---------------|
> | **macOS** | LaunchAgent (`~/Library/LaunchAgents/`) | No |
> | **Linux** | systemd user unit (`~/.config/systemd/user/`) | No (linger for boot) |
> | **Windows** | Task Scheduler (`schtasks`) | No |
> 
> ```bash
> mercury service status     # Check if service is running
> mercury service uninstall  # Remove the system service
> ```
> 
> In daemon mode, Telegram becomes your primary channel — CLI is log-only since there's no terminal for input.
>
